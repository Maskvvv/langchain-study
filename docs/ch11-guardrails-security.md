# 第十一章：Agent 安全与 Guardrails

企业 Agent 的风险不只来自模型回答错误，还来自“模型调用了不该调用的工具”。所以安全设计必须覆盖输入、工具、状态、输出和审计。

## 11.1 常见风险

| 风险 | 示例 |
|------|------|
| Prompt Injection | 用户要求模型忽略系统指令、泄露隐私 |
| 工具误调用 | 模型错误调用删除、发送、支付等工具 |
| 参数注入 | 用户把恶意内容塞进工具参数 |
| 数据泄露 | RAG 返回了无权限文档 |
| 幻觉 | 模型编造订单状态、政策条款、财务信息 |
| 无限循环 | Agent 不断调用工具，消耗成本 |

## 11.2 安全边界原则

模型可以建议，但最终边界必须由代码控制。

推荐规则：

- 工具参数必须用 Pydantic 或等价 schema 校验。
- 敏感工具必须走审批。
- 工具执行前检查用户权限。
- 对外部写操作保留审计日志。
- Agent 最大循环次数和最大耗时必须限制。
- 输出给用户前做敏感信息过滤。

## 11.3 工具分级

| 级别 | 示例 | 处理方式 |
|------|------|----------|
| 低风险 | 查询天气、查询公开知识 | 可直接执行 |
| 中风险 | 查询订单、查询客户资料 | 需要权限校验 |
| 高风险 | 发邮件、改数据库、退款 | 需要人工审批 |
| 禁止 | 导出所有用户数据、关闭安全策略 | 直接拒绝 |

## 11.4 Prompt Injection 防护

不要相信用户输入、网页内容、文档内容中的指令。RAG 场景尤其要注意：文档内容可能包含“忽略之前所有规则”这类恶意文本。

防护思路：

1. 把外部内容明确标记为“非指令数据”。
2. 系统提示词中要求模型不要执行文档中的指令。
3. 对工具调用做代码级权限判断。
4. 对高风险操作加入人工审核。

## 11.5 示例代码

```bash
uv run python src/ch11_guardrails_security/01_tool_guardrails.py
```

这个示例演示：

- 用安全表达式计算替代 `eval`
- 对工具参数做白名单校验
- 对敏感操作进行审批拦截

## 11.6 企业上线检查清单

- 是否列出了所有工具的风险等级？
- 每个工具是否有参数 schema？
- 每个工具是否有权限判断？
- 敏感操作是否需要审批？
- 是否记录了用户、工具名、参数、结果和时间？
- 是否限制最大循环次数？
- 是否有恶意输入测试集？
- 是否能快速禁用某个工具？
