# 第零章：LLM 应用基础

在学习 LangChain 之前，先理解 LLM 应用的基础概念非常重要。LangChain 不是魔法，它只是把模型、提示词、工具、检索、状态管理这些能力组织得更工程化。

## 0.1 一次 LLM 调用发生了什么？

```
用户输入
  ↓
组装消息列表：System / Human / AI / Tool
  ↓
发送给模型 API
  ↓
模型根据上下文预测下一个 token
  ↓
返回文本、结构化结果或工具调用请求
```

关键点：模型本身不会记住你之前说过什么。所谓“多轮对话”，本质是应用层把历史消息再次发给模型。

## 0.2 Token

Token 是模型处理文本的基本单位。它不完全等于中文字符、英文单词或字节。

你需要关心 token 的三个原因：

1. **上下文窗口限制**：模型一次能看到的 token 有上限。
2. **成本**：输入和输出 token 通常都会计费。
3. **速度**：上下文越长，请求越慢。

## 0.3 消息角色

现代 Chat Model 通常使用消息列表作为输入：

| 角色 | 说明 |
|------|------|
| System | 设定模型的身份、规则、边界 |
| Human | 用户输入 |
| AI | 模型之前的回复 |
| Tool | 工具执行结果 |

在 Agent 中，`ToolMessage` 非常关键：模型先提出工具调用，应用执行工具，再把工具结果以 `ToolMessage` 的形式送回模型。

## 0.4 Temperature

`temperature` 控制输出随机性：

| 值 | 特点 | 适合场景 |
|----|------|----------|
| 0 | 更稳定、更确定 | 分类、抽取、工具参数生成 |
| 0.2-0.5 | 稳定但不死板 | 问答、RAG、代码解释 |
| 0.7-1 | 更发散 | 创意写作、头脑风暴 |

企业 Agent 一般不追求“有创意”，而是追求稳定、可控、可评测，所以多数节点会使用较低 temperature。

## 0.5 上下文窗口

上下文窗口是模型一次调用能看到的最大 token 数。它包含：

- system prompt
- 历史对话
- 用户当前问题
- 检索出来的文档
- 工具调用历史
- 模型即将生成的输出

RAG、Memory、摘要、滑动窗口，都是为了解决上下文窗口有限和成本增长的问题。

## 0.6 结构化输出

企业应用很少只需要一段自然语言。很多时候你需要模型输出确定的结构：

```json
{
  "intent": "query_order",
  "order_id": "A123",
  "need_human_review": false
}
```

结构化输出常用于：

- 意图识别
- 路由决策
- 工具参数生成
- 审批判断
- 信息抽取

LangChain 中常用 `with_structured_output()` 配合 Pydantic 模型实现。

## 0.7 LLM 应用和普通后端的区别

| 普通后端 | LLM 应用 |
|----------|----------|
| 输入输出确定 | 输出有概率性 |
| 逻辑由代码控制 | 部分逻辑由模型判断 |
| 单元测试较稳定 | 需要评测集和统计指标 |
| 错误多是异常 | 错误可能是幻觉、误调用、漏调用 |
| 日志记录请求即可 | 需要记录 prompt、tool call、token、trace |

这也是为什么企业级 Agent 需要 LangGraph、LangSmith、Guardrails 和评测体系。

## 0.8 本项目中的学习顺序

建议你按下面顺序走：

```
LLM 基础 → Model I/O → Prompt / Parser → RAG → LCEL → Memory
        → Tool Calling → Agent → LangGraph → 企业级工程化
```

不要急着一上来写复杂 Agent。真正可靠的 Agent，往往是很多简单、明确、可测试的步骤组合出来的。
