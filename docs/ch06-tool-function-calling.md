# 第六章：Tool & Function Calling

## 6.1 为什么需要 Tool？

LLM 虽然强大，但有明显的局限：

| 局限 | 示例 |
|------|------|
| 无法获取实时信息 | "今天天气怎么样？" |
| 无法执行计算 | "12345 × 67890 = ?" |
| 无法访问外部系统 | "帮我查一下订单状态" |
| 无法执行操作 | "帮我发一封邮件" |

**Tool（工具）** 让 LLM 能够与外部世界交互，突破这些限制。

## 6.2 Function Calling 原理

Function Calling 是 LLM 的一种能力：模型可以**决定调用哪个函数**，并**生成函数参数**。

```
用户: "北京今天天气怎么样？"
         ↓
    LLM 分析意图
         ↓
    决定调用: get_weather(city="北京")
         ↓
    应用执行函数，获取结果
         ↓
    将结果返回给 LLM
         ↓
    LLM 生成最终回答: "北京今天晴天，气温25°C"
```

**关键点**：LLM 本身**不执行**函数，它只是生成函数调用的指令。实际执行由应用层完成。

## 6.3 LangChain 中的 Tool

LangChain 提供了统一的 Tool 抽象：

```python
from langchain_core.tools import tool

@tool
def search_weather(city: str) -> str:
    """查询指定城市的天气"""
    # 实际实现...
    return f"{city}今天晴天，25°C"
```

### Tool 的核心要素

| 要素 | 说明 |
|------|------|
| **名称** | 函数名，LLM 通过名称选择调用哪个工具 |
| **描述** | 函数的功能描述，LLM 根据描述判断何时调用 |
| **参数 Schema** | 函数参数的 JSON Schema，LLM 据此生成参数 |
| **实现** | 函数的实际逻辑 |

> ⚠️ **描述非常重要**！LLM 完全依赖描述来决定是否调用工具。描述不清晰 = 调用错误。

## 6.4 结构化输出

Function Calling 本质上是一种**结构化输出**——让 LLM 输出符合预定义 Schema 的 JSON，而非自由文本。

LangChain 还提供了 `with_structured_output()` 方法，直接将 LLM 输出解析为 Pydantic 模型。

## 6.5 多工具协作

当 LLM 绑定了多个工具时，它可以：

1. **选择合适的工具**：根据用户问题选择最相关的工具
2. **连续调用多个工具**：先调 A 获取信息，再调 B 执行操作
3. **并行调用**：同时调用多个独立工具（部分模型支持）

## 6.6 原理深入：Tool Calling 的底层流程

```
1. 应用将工具定义（名称、描述、参数Schema）随消息一起发送给 LLM API
2. LLM 判断是否需要调用工具
   - 不需要：直接返回文本回复
   - 需要：返回 tool_calls 字段，包含工具名和参数
3. 应用解析 tool_calls，执行对应的函数
4. 将函数结果作为 ToolMessage 追加到消息列表
5. 再次调用 LLM，让它基于工具结果生成最终回答
```
