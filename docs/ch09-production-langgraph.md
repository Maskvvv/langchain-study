# 第九章：企业级 LangGraph

第八章已经学习了 StateGraph、条件路由、人机协作和 Multi-Agent。本章开始关注生产环境：如何让 Agent 可恢复、可审计、可重试、可控制。

## 9.1 为什么生产环境更适合 LangGraph？

传统 Agent 往往是“模型自由循环调用工具”，优点是灵活，缺点是难以控制。企业场景更看重：

- 每一步可解释
- 每个工具调用可审计
- 失败后可恢复
- 敏感操作可暂停审批
- 长任务可以跨请求继续执行
- 可以针对节点做测试和评测

LangGraph 的核心价值就是把 Agent 从“黑盒循环”变成“显式状态图”。

## 9.2 企业级 Agent 的基本结构

```
用户请求
  ↓
意图识别节点
  ↓
权限与安全检查节点
  ↓
检索 / 工具调用 / 计划执行
  ↓
人工审批节点（可选）
  ↓
结果生成节点
  ↓
日志、评测、监控
```

不是所有任务都需要 Agent。能用固定 Chain 解决的任务，就不要交给 Agent 自由决策。企业级设计的原则是：让模型负责语义判断，让代码负责边界和流程。

## 9.3 Checkpoint

Checkpoint 用于保存图执行过程中的状态。它解决三个问题：

1. **多轮会话**：同一个 `thread_id` 下恢复历史状态。
2. **断点续跑**：流程暂停后继续执行。
3. **故障恢复**：节点失败后可以从最近状态重试。

示例代码见：

```bash
uv run python src/ch09_production_langgraph/01_checkpoint_thread.py
```

## 9.4 Human-in-the-Loop

企业 Agent 中，下面这些操作通常不能让模型直接执行：

- 发邮件、发通知
- 删除或修改生产数据
- 创建订单、退款、转账
- 调用外部系统写接口
- 访问敏感客户信息

推荐做法：

1. 模型只生成“待执行计划”。
2. 代码判断是否敏感。
3. 敏感操作进入人工审批节点。
4. 审批通过后由后端执行工具。
5. 记录完整审计日志。

## 9.5 错误处理与重试

生产环境需要区分不同错误：

| 错误类型 | 处理方式 |
|----------|----------|
| 模型超时 | 重试或降级模型 |
| 工具参数错误 | 让模型修正参数，限制次数 |
| 外部 API 错误 | 指数退避、熔断、告警 |
| 权限不足 | 不重试，返回权限提示 |
| 输出格式错误 | 结构化解析重试 |

不要无限循环。每个 Agent 都应该设置最大步数、最大耗时和失败出口。

## 9.6 节点设计原则

一个好的 LangGraph 节点应该：

- 输入输出结构明确
- 只负责一类事情
- 可以单独测试
- 不偷偷修改全局状态
- 对外部系统调用有超时和错误处理
- 日志里能看出它为什么做了这个决策

## 9.7 生产化检查清单

- 是否有 `thread_id` 和 checkpoint？
- 是否限制最大循环次数？
- 工具是否做了参数校验？
- 敏感工具是否需要审批？
- 是否能追踪每次模型和工具调用？
- 是否有评测集防止 prompt 修改后质量下降？
- 是否能统计 token 成本和耗时？
- 是否有失败降级策略？
